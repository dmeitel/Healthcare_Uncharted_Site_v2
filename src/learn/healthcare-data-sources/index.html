Infrastructure Data sourcing

Where the data lives.

Every map and tool on this site runs on real public data. Not vendor decks, not vibes. Real federal and nonprofit datasets, pulled, cleaned, and housed so that one number can show up in five places at once. This is how that works, and where every piece of it comes from.

Principle The hard part of a data project was never the chart. It is knowing where the number came from, and whether you can get it again next year. Provenance first. Pretty second.

The premise

The chart is the easy part.

People think the work in a data project is the visual. The colors, the hover states, the animation. That is the last ten percent, and it is the fun part, so it gets all the attention. The real work happens before any of that. It is finding the number, trusting the number, and being able to get the number again when a fresh version drops eighteen months from now.

Most healthcare dashboards you have seen fail that last test. Somebody pulled a spreadsheet once, pasted the values into a slide, and shipped it. Six months later nobody remembers which file it came from or how it was cleaned. The chart still renders. It is just quietly wrong, and getting wronger.

"A number with no source is a rumor with a font."

So this site treats sourcing as the actual product. Before a metric shows up on a map, it carries its origin with it. What agency published it, the exact dataset, the year it represents, the method behind it, and the caveats that keep an honest person from over-reading it. The visual is downstream of all that. This page is the tour of the part nobody usually shows you.

What is relevant and sourceable

The shelves, and what fills them.

Healthcare is not one dataset. It is six or seven different worlds that only make sense together. Population health on its own is trivia. Population health next to insurance coverage, provider supply, and household income starts to explain something. So we pull across all of it, and we group what we pull the way the system actually breaks down.

Why these sources, and not others

There is a lot of healthcare data for sale. Ranking sites, scorecards, private aggregators with slick dashboards. We do not use them, and the reason is a rule, not a preference. A source has to pass four tests to earn a spot on the shelf. Almost all the paid aggregators are built on top of these same federal feeds anyway. We just go to the source instead of the reseller.

system of record
The agency that actually defines the measure, not someone repackaging it. CDC for disease prevalence. CMS for facilities. Census for population and income. The original, not the recap.
public + reproducible
Free, no license, no paywall. You can pull the exact same file we did and check our work. Credibility you can audit beats credibility you have to trust.
standard geography
It keys to FIPS, state, or county. A source that uses its own made-up regions cannot join the shelf, so it cannot sit next to anything else. No geography, no entry.
uniform coverage
All 50 states and DC, measured the same way. Not a 12-state sample dressed up as national. If a state is missing, we say so out loud.

How we house it

One shelf. No islands.

Here is the move that makes the whole thing work. Every dataset, no matter how different it looks, gets flattened into one shape and filed under one shared key. A hospital and a poverty rate have nothing in common on the surface. But both of them happen in a place. So both get filed under the state. That shared shelf is what lets them be read together.

Sources in, one shelf, tools out
The pipeline
Update the shelf once, the whole storefront moves. No tool stores its own copy of the data. They all read the same shelf. So a corrected hospital count or a fresh unemployment rate shows up everywhere it is used, the moment the warehouse rebuilds.

Everything is the same shape

A hospital, a nursing role, a state, a policy, a chronic-disease rate. Underneath, they are all the same record. That sameness is the whole trick. The plumbing never has to know what kind of thing it is holding, which means adding a brand new dataset is never a rebuild. It is just one more thing filed into the same shape.

uid
A unique, namespaced id. This is the join key. A hospital "in Utah," a poverty rate "for UT," and a policy "in Utah" all resolve to the same place, so they meet on the shelf.
type + label
What kind of thing it is, and its human name.
facets
The structured, filterable detail. Ownership, beds, the unit, the year, whether higher is better.
search
A precomputed text string, so the same record is findable everywhere without re-indexing.
rels
Typed links to other records, with the value carried on the link itself. "This metric describes Utah, value 8.2%."
source
Where it came from. The dataset, the agency, the retrieval date. Provenance travels with the data, not in a footnote.

Because the value rides on the link between a metric and its state, the same slot can later carry weight, direction, and the source of a claim like "this policy moved that metric." That is a field we add, not a rebuild we suffer. The shape was chosen so the painful retrofits never happen.

The payoff
Once everything shares the shelf, the islands disappear. A respiratory therapist role links to the metric that counts RTs per capita, and that metric already describes every state, which is where the hospitals and the suppliers already live. Follow the links and you walk from a single job title all the way to the map. That walk is only possible because nothing was stored off on its own.
Keeping it honest

New data drops. Now what?

Public data is not static. CDC, Census, and CMS push fresh releases on their own schedules, usually every six to twelve months, often with a two-year lag baked in. A site built on that data is only as trustworthy as its plan for the day the numbers change. Without a plan, every dataset slowly rots in place while the chart keeps rendering like nothing happened.

So there are two refresh modes here, and the site is honest about which is which. Some layers are scripted: a pull script hits the source, cleans it, and rewrites the file. The facility maps work this way. Re-run, review the diff, done. Other layers are hand-compiled today, with the vintage of every single metric tracked so the site always knows how old each number is. The direction is to give every source its own script, so a refresh is never a research project.

01
Pull
A script per source fetches the latest release straight from the agency.
02
Clean
Normalize, geocode, file under the shared state key, attach the source tag.
03
Validate
Check the shape. Flag broken links and any source whose format quietly changed.
04
Diff
Show what moved before anything ships. No blind overwrites.
05
Rebuild
One rebuild, and every tool on the shelf reflects the new numbers at once.

Two rules keep that loop trustworthy. Staleness is a fact we show, not a thing we hide. Every metric carries its vintage, and old numbers say so out loud instead of pretending to be current. And sources drift. Agencies move URLs, change column names, start requiring keys. The pull has to notice and flag that, not silently write garbage into a clean file. A refresh that fails loudly beats a refresh that lies quietly.

What we can promise

Honest beats clean.

Here is a claim I will not make: that this data is perfect. Nobody who works with public health data gets to say that and mean it. The numbers come from surveys, models, and estimates, and they carry every limit those methods carry. What I will claim is narrower and a lot more useful. The data is honest. You can see exactly where every number came from, how it was handled, and how old it is. Four things we stand behind, and two we will never pretend.

provenance
Every value traces to a named dataset, its ID, its URL, and the date we pulled it. No orphan numbers. If you want to check us, the link is right there in the metric.
faithful
We do not hand-edit values. Where we transform one, like weighting county prevalence up to a state, it is open, deterministic code you could re-run. Our number is a documented function of the official one, not a guess.
validated
Every refresh checks that all 51 states resolve, that counties match real FIPS codes, that no link points at nothing. The pull refuses to write when something unexpected goes missing. That is what clean actually means here: complete and sound, not silently broken.
dated
Every metric carries its vintage and shows it. Old numbers say they are old instead of pretending to be current. You always know how fresh the thing you are looking at is.
What we will not pretend
We do not control the accuracy of the source itself. CDC PLACES is modeled from a self-reported survey, with confidence intervals and the occasional methodology change that moves every state at once. Obesity did exactly that in 2022. We inherit those limits, so we carry the source's own caveats instead of burying them. And numbers from different agencies, different years, and different methods are for seeing the shape of a thing, not for splitting hairs at the decimal. Directional truth, honestly sourced. That is the standard, and it is a higher one than clean.
Why it is built this way

One dataset, many windows.

The reason for all of this is leverage. When the data lives underneath instead of inside any one tool, every tool gets to reuse it, and every new tool starts with the whole warehouse already behind it. The same hospital file drives the operations map, feeds state rollups, and answers a search in the Atlas. Build once, surface everywhere.

The short version

Pull real public data. File it under one shared key so nothing ends up an island. Carry its source and its vintage with it, always. Then let every tool read from the same shelf, so updating the data once updates the whole site. That is the entire architecture. Everything else is just more shelves.
